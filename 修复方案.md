## Linux 部署修复方案

### Bug 1: `timeout: command not found`

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | 大概率不会。`timeout` 是 GNU coreutils 的一部分，主流 Linux 发行版（Ubuntu/Debian/CentOS）都预装 |
| **修复方案** | 如果极简系统（如 Docker alpine）缺失，安装 `coreutils`：<br>Ubuntu/Debian: `apt install coreutils`<br>CentOS/RHEL: `yum install coreutils` |

---

### Bug 2: macOS 下 `Darwin` 未被加入软链创建判断

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **不会**。`experiment.py:119` 原本就是 `if platform.system() == "Linux"`，Linux 始终在支持范围内 |
| **修复方案** | 无需操作 |

---

### Bug 3: 子进程路径重复拼接

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **会**。这是平台无关的逻辑 bug，相对路径 + 相对 cwd 在任何 OS 上都会出现路径重复 |
| **修复方案** | 同样使用 `.absolute()`：<br>• `quantaalpha/factors/coder/factor.py:179`<br>  `execution_code_path` → `execution_code_path.absolute()`<br>• `quantaalpha/factors/runner.py:119`<br>  `ws.workspace_path / 'factor.py'` → `(ws.workspace_path / 'factor.py').absolute()` |

---

### Bug 4: `CONDA_DEFAULT_ENV` 为 None 导致 CondaConf 验证失败

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **会**。只要启动后端/子进程时 conda 环境未激活或环境变量未传递，就会触发 |
| **修复方案** | 修改 rdagent 中 `factor_coder/config.py` 和 `utils/env.py`（两个 Python 环境都需要改）：<br>`os.environ.get("CONDA_DEFAULT_ENV")` → `os.environ.get("CONDA_DEFAULT_ENV") or os.environ.get("CONDA_ENV_NAME", "quantaalpha")`<br>确保 `.env` 文件中有 `CONDA_ENV_NAME=quantaalpha` |

---

### Bug 5: Embedding API 404

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **会**，只要使用的 LLM 后端不支持 embedding 接口 |
| **修复方案** | 1. 在 `.env` 中设置 `EMBEDING_MODEL=` 置空，跳过 embedding<br>2. `quantaalpha/llm/client.py` 中 `create_embedding` 方法检查 model 为空时返回零向量<br>3. `calculate_embedding_distance_between_str_list` 添加 try/except 兜底 |

---

### Bug 6: Qlib 数据软链使用相对路径

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **不会**。Linux 原生支持软链，`os.symlink` 创建相对路径软链的行为与 macOS 一致。但如果部署时 `QLIB_DATA_DIR` 设为相对路径且 CWD 变化，同样会出问题 |
| **修复方案** | `frontend-v2/backend/app.py:217-225` — 创建软链前用 `PROJECT_ROOT / qlib_data_path` 转为绝对路径（已修复） |

---

### Bug 7: LightGBM 找不到 `libomp.dylib`

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **可能会**，但原因不同。Linux 上 LightGBM 需要 `libgomp.so`（GCC OpenMP）或 `libomp.so`（LLVM OpenMP），取决于 LightGBM 的编译方式 |
| **修复方案** | <br>• conda 安装的 LightGBM：`conda install -c conda-forge llvm-openmp`<br>• 系统安装的 LightGBM：`apt install libgomp1`（Ubuntu/Debian）或 `yum install libgomp`（CentOS）<br>• 验证：`python -c "from lightgbm import LGBMClassifier; print('OK')"` |

---

### Bug 8: MLflow 文件存储后端已废弃

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **会**，同样触发 `MlflowException: The filesystem tracking backend is in maintenance mode` |
| **修复方案** | 在 `.env` 中添加：`MLFLOW_ALLOW_FILE_STORE=true`<br>或设置环境变量：`export MLFLOW_ALLOW_FILE_STORE=true` |

---

### Bug 9: `CondaConf.bin_path` 为空

| 维度 | 说明 |
|------|------|
| **Linux 上是否会遇到** | **可能会**。如果 conda 通过 `conda init` 注册为 shell 函数，`subprocess.run(shell=True)` 使用 `/bin/sh` 时同样找不到 `conda` 命令 |
| **修复方案** | 修改 rdagent `utils/env.py` 中 `change_bin_path` 方法：<br>• 优先使用 `$CONDA_EXE` 绝对路径（如 `/opt/conda/bin/conda`）<br>• 失败时 fallback 到 `$CONDA_PREFIX/bin:$PATH`<br>• 两个 Python 环境的 rdagent 都要改 |

---

 总结

| Bug | Linux 是否需要修复 | 备注 |
|-----|----------------|------|
| #1 timeout | 大概率不需要 | Linux 预装 |
| #2 Darwin 软链 | 不需要 | Linux 本就支持 |
| #3 路径重复 | **需要** | 平台无关 |
| #4 CondaConf 验证 | **需要** | 平台无关 |
| #5 Embedding 404 | **需要** | 平台无关 |
| #6 Qlib 软链路径 | 建议修复 | 保持健壮性 |
| #7 LightGBM libomp | **需要** | 不同依赖名 |
| #8 MLflow | **需要** | 平台无关 |
| #9 bin_path 为空 | **需要** | 平台无关 |

实际需要在 Linux 上修复的是 **3、4、5、7、8、9** 这 6 个 bug，其中 #7 的修复命令略有不同（`libgomp1` vs `llvm-openmp`）。vm-openmp`）。

## 前端启动挖矿任务时，我用的参数如下：

| 字段 | 值 |
|------|-----|
| 研究方向 | `价量关系与开盘收益率` |
| 并行方向数 | 2 |
| 进化轮次 | 1 |
| 迭代次数 | 1 |
| 每假设因子数 | 1 |
| 质量门控 | 关闭 |
| 并行执行 | 关闭 |

对应在前端界面的操作：

1. 研究方向输入框填：**价量关系与开盘收益率**
2. 并行方向数设为 **2**
3. 进化轮次设为 **1**
4. 迭代次数设为 **1**
5. 每假设因子数设为 **1**
6. 质量门控和并行执行保持默认关闭

这样设置大约 **10-15 分钟**能跑出一轮结果，产出 3-8 个带真实回测指标的因子。如果想要更多因子，可以把进化轮次和迭代次数调大（比如各设为 2-3），但运行时间会相应增加。
